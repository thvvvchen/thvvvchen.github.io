TCP/UDP：两者的区别及适用场景

项目 TCP UDP
连接方式 面向连接（3 次握手） 无连接
可靠性 可靠传输：有确认机制、超时重传、丢包重发、乱序重排 不可靠：不保证到达、不重传、不排序
传输顺序 严格保证顺序（按序到达） 不保证顺序
速度 慢（建立连接、确认、拥塞控制） 快（无握手、无确认）
开销 大（header 20 字节以上） 小（header 8 字节）
拥塞控制/流量控制 有（避免网络阻塞） 无（尽可能快发送）
应用层调试难度 高 低

一句话总结：

TCP：可靠但慢；UDP：快速但不可靠。

适用场景:

TCP 可靠传输的手段

1. 确认应答机制（ACK）

  • 每收到一段数据，接收端会返回一个 ACK 确认号。

2. 超时重传（Timeout Retransmission）

  • 发送端在发出数据后启动重传定时器（RTO）。
  • 若在超时时间内没收到 ACK → 自动重传数据包。

3. 序列号与数据按序重组

  • TCP 使用 序列号（Sequence Number） 标识数据顺序。
  • 即使数据乱序到达，接收端也能根据序号 重新排序。

4. 流量控制（Flow Control）

  • 在 TCP 里：
  • 每个 ACK 报文里都有一个 Window 字段（接收窗口 rwnd）

​ “从我这边已经收到的数据之后起，我还能再接收多少字节。”

发送端看到这个值，就会限制自己已发送但未被确认的数据量不超过这个窗口。

这套机制，本质就是一个 滑动窗口协议：

  • 窗口的“左边”是已确认的数据
  • 窗口的“中间”是已发送但未确认的数据
  • 窗口的“右边”是可以继续发送的空间
  • 每当收到 ACK，窗口就向前滑动

👉 所以可以说：TCP 流量控制是通过接收端通告的滑动窗口实现的。

5. 拥塞控制（Congestion Control）

包括几个算法：

  • 慢启动（Slow Start）
  • 拥塞避免（Congestion Avoidance）
  • 快速重传（Fast Retransmit）
  • 快速恢复（Fast Recovery）

为什么是三次握手四次挥手？

握手：

💡如果没有第三次握手会怎样？（只两次）

  • 服务端发出 SYN+ACK 却没收到 ACK：

    • 它并不确定客户端是不是“还活着”、网络是不是中途挂了
    • 如果直接认为连接建立，就可能出现“半开连接”、资源白占

👉 所以： 两次不够确认双方状态，三次是确保双方都“明确在线、序列号同步”的最少次数。

挥手：

✅ TCP 是全双工的

  • 一条 TCP 连接里，有两条独立的方向：

    • 我 → 你
    • 你 → 我

  • 关闭时，并不一定两个方向同时不用了：

    • 可能我发完数据了，不想再发，但你还在发给我，也就是确保双方都发送完数据了，并且都要关闭。
  1. 第一次挥手：A → B，FIN

“我这边不再给你发数据了（但还能收），准备关发送方向。”

  1. 第二次挥手：B → A，ACK

“收到，你不发了我知道了，但我这边可能还有数据没发完。”

    • **此时：**A 不能再发数据，但还能收；B 还能发还能收。
  1. 第三次挥手：B → A，FIN

“我这边也发完了，也不再给你发了。”

  1. 第四次挥手：A → B，ACK

“收到，你也不发了，那咱俩都关了。”

💡为什么不能合并成三次？

**有时第三次和第二次可以合并（ACK + FIN 同一个报文）， **那就是“三次挥手”的变种（一次性确认+发FIN），但这是“可以优化”的情况，不是协议要求。

协议设计的是：

**两个方向独立关闭，每个方向都是 FIN + ACK。 **标准情况是四次，这是最通用、最保险的做法。

4开头状态码

4开头状态码

400 Bad Request
请求格式有问题，比如：

    • JSON 语法错了
    • 必填参数没传
    • URL 太乱，服务器解析不了
  • 401 Unauthorized
    未认证/未登录：
    • 需要带 token / 需要登录，但你没带或带错了
  • 403 Forbidden
    已经认证了，但没有权限：
    • 你是普通用户，访问了管理员接口
    • IP 被拉黑
  • 404 Not Found
    资源不存在：
    • URL 写错
    • 接口路径没这个路由
    • 静态文件没找到
  • 405 Method Not Allowed
    请求方法不被允许：
    • 后端只支持 GET，你发了 POST
    • 只允许 POST，你用了 PUT
  • 408 Request Timeout
    请求超时：
    • 客户端请求太慢 / 连接挂太久没发完
  • 409 Conflict
    资源状态冲突：
    • 比如并发修改同一条数据导致版本冲突
  • 429 Too Many Requests
    请求太频繁：
    • 被限流了（例如接口 QPS 超过阈值）


强缓存和协商缓存

浏览器缓存？http缓存协议？

浏览器缓存是浏览器对已请求资源的本地存储机制，核心分为强缓存和协商缓存，由 HTTP 缓存协议（HTTP 头字段）控制。

一、强缓存

浏览器直接从本地缓存读取资源，不向服务器发起请求，由以下 HTTP 头控制：

  1. Expires（HTTP/1.0）
    • 格式：Expires: Wed, 26 Dec 2025 12:00:00 GMT
    • 含义：资源的过期时间（服务器时间），若当前时间早于 Expires，直接用缓存。
    • 缺陷：依赖服务器与客户端时间同步，易因时间偏差失效。
  1. Cache-Control（HTTP/1.1，优先级更高）
    • 常用值：
      • max-age=3600：资源在本地缓存 3600 秒（从请求成功时开始计时）。
      • public：允许任何缓存（浏览器、CDN 等）存储该资源。
      • private：仅允许浏览器（私有缓存）存储，禁止 CDN 等共享缓存。
      • no-cache：不使用强缓存，直接进入协商缓存。
      • no-store：禁止任何缓存，每次都从服务器获取新资源。

二、协商缓存

浏览器先向服务器发送请求，由服务器判断资源是否更新：若未更新，返回 304 状态码，浏览器用本地缓存；若已更新，返回 200 和新资源。由以下 HTTP 头配对控制：

  1. Last-Modified & If-Modified-Since
    • 服务器响应头：Last-Modified: Tue, 25 Dec 2025 10:00:00 GMT（资源最后修改时间）。
    • 浏览器请求头：If-Modified-Since: Tue, 25 Dec 2025 10:00:00 GMT（携带上次的 Last-Modified 值）。
    • 服务器逻辑：对比资源当前修改时间与 If-Modified-Since，若未变化则返回 304。
    • 缺陷：无法识别 “文件内容未变但修改时间变化” 的情况，精度仅到秒级。
  1. ETag & If-None-Match（优先级更高）
    • 服务器响应头：ETag: "abc123"（资源内容的唯一标识，通常是哈希值）。
    • 浏览器请求头：If-None-Match: "abc123"（携带上次的 ETag 值）。
    • 服务器逻辑：对比资源当前 ETag 与 If-None-Match，若一致则返回 304。
    • 优势：基于内容标识，精度更高，避免时间偏差问题。

三、缓存流程总结

  1. 浏览器请求资源时，先检查Cache-Control/Expires：若未过期，直接用强缓存。
  2. 若强缓存过期，发起请求并携带If-Modified-Since/If-None-Match，进入协商缓存。
  3. 服务器判断资源是否更新：
    • 未更新：返回 304，浏览器用本地缓存。
    • 已更新：返回 200 + 新资源，更新本地缓存。

lastmodified是请求一次还是两次？

一次，强缓存过期之后，浏览器会携带if…..，请求服务器，服务器比对last….，返回304或者200以及新数据。

last和etag的区别？

last：精确度是秒级的，无法识别秒内对内容的修改，他的优先级也比etag更低，更适用于对内容变化敏感度低、修改频率低的资源。

etag：是资源内容的唯一哈希值，精确度更高，优先级更高，适用于那些对内容变化敏感的场景。但是由于哈希计算，会额外增加服务器计算成本。

网络攻击和攻击的方式

1. 跨站脚本攻击（XSS）

  • 为什么需要了解：XSS 攻击是针对前端的常见攻击，攻击者通过在网页中注入恶意的 JavaScript 代码，窃取用户的敏感信息（如 Cookies、token）或篡改页面内容。

  • 有哪些类型：

  • 反射型 XSS：恶意脚本通过 URL 传入，服务器直接返回给用户执行。

  • 存储型 XSS：恶意脚本被存储在服务器（如数据库、文件）中，用户访问时自动执行。

  • DOM-based XSS：攻击者通过修改浏览器的 DOM 内容，直接执行恶意脚本。

  • 如何防范：

    • 对所有用户输入进行 HTML 转义。
    • 使用 Content Security Policy (CSP) 来限制外部资源加载。
    • 不直接将用户输入直接插入 DOM，避免内联 JavaScript。

2. 跨站请求伪造（CSRF）

  • 攻击方式：攻击者诱使用户在已登录的情况下，点击恶意链接或加载恶意页面，自动向服务器发起请求。

  • 如何防范：

    • 在表单提交中使用 CSRF Token，确保请求来源合法。
    • 采用 SameSite Cookie 属性，防止跨站请求伪造。

3. 网络钓鱼（Phishing）

  • 攻击方式： 钓鱼攻击通过伪装成合法网站或应用诱导用户输入个人信息，前端需要时刻关注用户交互界面的安全性，防止用户误入假网站、点击恶意链接。

  • 如何防范：

    • 通过 HTTPS 协议确保用户与服务器之间的数据加密传输。
    • 教育用户识别 假网站，在 UI 中使用明显的身份验证提示。
    • 在前端输入表单和链接上使用安全标识，确保所有外部链接都是可信的。
